# AGENTS.md

This file provides guidance to Codex (Codex.ai/code) when working with code in this repository.

Tool that turns a YouTube URL into a Markdown summary. It does **not** use any LLM HTTP API — providers shell out to a locally-installed CLI driven by the user's own subscription and keep only the model's final message. Two providers ship: `codex` (the `codex` CLI, default) and `Codex` (the `Codex` / Codex CLI, via `Codex --print`). The README is in Russian; this file and the code are in English.

The entire backend lives in the local SwiftPM package **`TubeFoldKit/`**; the macOS app runs it in-process and a small `tubefold` CLI target reuses it.

## Commands

```bash
# All tests (~130, Swift Testing; the canonical runner)
cd TubeFoldKit && swift test

# One suite / one test
swift test --filter SummaryPipelineTests
swift test --filter TranscriptSelectionTests/manualOriginalPreferredOverAuto

# Exercise the full pipeline WITHOUT a model call — use this for fast iteration
swift run tubefold "https://youtu.be/dQw4w9WgXcQ" --provider fake --no-open

# Live InnerTube smoke test (metadata + caption tracks + transcript, JSON out)
swift run tubefold-harness "https://youtu.be/dQw4w9WgXcQ" --transcript

# Build the macOS app from the terminal (from the repo root)
xcodebuild -project "TubeFold App/TubeFold.xcodeproj" \
  -scheme "TubeFold" -configuration Debug -destination "platform=macOS" build

# Install the CLI (builds release binary, symlinks into ~/.local/bin)
./install.sh
```

## Architecture

There are **two front ends over one shared Swift package** (`TubeFoldKit/`):

- **The macOS app** (`TubeFold App/`) — SwiftUI; runs the backend **in-process** via `TubeFoldBackend.shared` (no helper process, no localhost round-trips for its own UI) and serves the Chrome extension over a tiny localhost HTTP shim.
- **`tubefold` CLI** (`TubeFoldKit/Sources/tubefold-cli/`) — one-shot, synchronous: URL in, saved `.md` path on stdout, diagnostics on stderr. Config layering, lowest to highest: built-in defaults → `~/.config/tubefold/config.env` (or `--config` / `$TUBEFOLD_CONFIG`) → environment variables → CLI flags. The older config keys `ALLOW_ANY_SUBTITLE_LANGUAGE`/`SUMMARY_LANGUAGE` still map onto `ALLOW_ANY_TRANSCRIPT_LANGUAGE`/`OUTPUT_LANGUAGE`.

Pipeline shape (same for both):

```
URL → parse video id → InnerTube player (metadata + caption tracks)
    → timedtext download → render prompt from bundled template → provider
    → strip fence + validate → YAML front matter + body
```

Key invariant: **the YAML front matter is generated by the pipeline, never by the model.** The provider only ever produces the Markdown body, which is appended after the front matter.

### TubeFoldKit layout (`TubeFoldKit/Sources/TubeFoldKit/`)

| Area | What lives there |
|---|---|
| `Core/` | Pure logic: `YouTubeURL` (id/URL parsing), `Filenames` (safe/unique names, `[TubeFold] …` artifacts), `SummaryText` (front matter, fence stripping, validation, footer, prompt templating), `OutputLanguage`, `EnvFile` |
| `InnerTube/` | `InnerTubeClient` — metadata + transcripts straight from YouTube's InnerTube API. One `POST youtubei/v1/player` per client profile; **multi-client fallback ANDROID → iOS → TVHTML5** for the `exp=xpe` class of blocks; a WEB-client call only to enrich the publish date (mobile clients omit `microformat`). Client name/version constants live in **one file** (`InnerTubeProfiles.swift`) — they rot, bump them there. Transcript download hits the track's signed `baseUrl`; the response is timedtext **XML** (`fmt=json3` is ignored on signed URLs), parsed by sniffing (`json3` accepted too). Track selection: infer the **original spoken language** from the ASR track, prefer a manual track within it; `ALLOW_ANY_TRANSCRIPT_LANGUAGE` only governs the no-ASR fallback. Failed transcript **is** fatal; metadata is best-effort (video-id stub fallback), never fatal. |
| `Providers/` | `CodexProvider` / `ClaudeProvider` — direct `Process` invocation (`codex exec` from an isolated temp cwd with `--sandbox read-only --ephemeral --ignore-rules --skip-git-repo-check`, prompt on stdin, final message via `--output-last-message`; `Codex --print --output-format json`, isolated cwd so no AGENTS.md discovery, subscription auth). Token usage is parsed **in-process** (`UsageParsing`: both Codex JSONL schemas — flat `turn.completed` and legacy `token_count` — plus the Codex result object; best-effort, never fails a job). `ScriptProvider` keeps the classic `provider <prompt_file> <output_file>` executable contract as an escape hatch; `FakeProvider` is the test double. `ProviderDiagnostics` + `ProviderSetupStore` (state in `provider-setup.json`, provider-prefixed keys, also holds the `outputLanguage` app setting) drive onboarding: detect binary → marker connection test → persist state (never credentials). |
| `Store/` | SwiftData: `@Model` `Video`/`Job`/`WatchActivity`/`AppMeta` behind the `@ModelActor` **`VideoStore`** (`library.store` in the data dir). All the old repository semantics: `createOrReuse` dedupe (`force_regenerate` overrides; a `failed` video re-queues without force), orphaned-job reclaim on launch, watch-activity upsert that **preserves dismissals**, `latestWatchSuggestion` skipping library videos, `usageSummary`, `reset` (rows only — provider setup and Telegraph account survive). Model objects never cross the actor boundary; methods return `Sendable` snapshot structs. |
| `Pipeline/` | **`SummaryPipeline` actor** — single-flight queue draining `queued` jobs one at a time through the same stages/statuses as ever (`fetchingMetadata → fetchingTranscript → generatingSummary → ready/failed`), writing per-job artifacts (`jobs/<id>/…` incl. `job.log`) and final copies to `videos/<youtube_id>/`. Which provider runs: the UI selection in `provider-setup.json` for codex/Codex, the configured name verbatim otherwise (e.g. `fake`). Reasoning effort is always `"auto"` (the CLI picks the model default). |
| `Telegraph/` | `TelegraphMarkdown` (Markdown → Telegraph nodes; h1/h2 → h3/h4), `TelegraphClient` (injectable transport), `TelegraphPublisher` — one public page per video: repeat publish with unchanged summary reopens the URL, a regenerated summary edits the same page. Account token in `telegraph-account.json` (created once, kept on reset). Pages are public and can't be deleted. Content capped under 64 KB with a truncation notice. |
| `ReadingTime/` | Port of the `readtime` algorithm (265 WPM, image decay 12s→3s, `max(1, ceil)`) over the Telegraph-parsed visible text. Surfaced as `readingTimeMinutes` per ready video and in the Telegraph credit line. |
| `Backend/` | **`TubeFoldBackend`** — the facade everything talks to (`TubeFoldBackend.shared` in the app). Owns store + pipeline + setup store + Telegraph, exposes the operations of the old HTTP API with payload dicts key-for-key identical to the old `server.py`; the app's typed models decode straight from them. **`ExtensionServer`** — `NWListener` + minimal HTTP/1.1 on `127.0.0.1:43821`, serving the Chrome-extension contract (see below). |
| `Resources/prompts/` | `detailed-summary.md` prompt template (`{{VAR}}` placeholders), bundled into both the app and the CLI. |

### macOS app (`TubeFold App/`)

`AppDelegate` calls `TubeFoldBackend.shared.startServing()` on launch (reclaims orphaned jobs, drains queued work, starts the extension shim) and `stopServing()` on quit. `LibraryService` / `ProviderSetupService` are thin wrappers translating facade payloads into the app's typed models — no HTTP, no health checks, no `backendFeatures` gate (client and backend are the same binary). The data dir is `~/Library/Application Support/TubeFold/` (`library.store`, `videos/`, `jobs/`, `logs/`, `provider-setup.json`, `telegraph-account.json`); a stale `database.sqlite` left by old builds is deleted on first launch.

### Extension HTTP API (the only HTTP surface)

`ExtensionServer` exists solely for the Chrome extension (separate repo, [TubeFold/extension](https://github.com/TubeFold/extension)). Contract verified against the extension's actual fetches: `GET /health` (`apiVersion: 1` + `backendFeatures` capability map — for the extension only), `POST /api/v1/summaries`, `POST /api/v1/watch-activity`, `GET /api/v1/videos/by-youtube-id/{id}`, `POST /api/v1/videos/{id}/publish-telegraph`; the wider library/settings surface (videos list/delete/regenerate, watch suggestion/dismiss, extension-status, usage, provider-setup, reset) is served for compatibility. CORS for `chrome-extension://*`, loopback bind only, optional `TUBEFOLD_API_TOKEN` bearer auth, error shape `{"error": {code, message}}`. A `chrome-extension://` Origin marks the extension as seen (throttled) — that powers the Settings "extension connected" tile. **If you change an endpoint the extension calls, update the extension repo too.**

## Notable constraints

- MVP scope: no playlists, no Whisper fallback, no chunked summarization (long transcripts are sent whole).
- The provider CLI (`codex` and/or `Codex`) is an external user dependency even in the packaged app. `install.sh` requires at least one of the two.
- InnerTube client versions are a maintenance burden: YouTube retires old ones. If transcript fetches start failing across all client profiles, check the current `clientVersion` values other InnerTube consumers ship (youtube-transcript-api, NewPipe) and bump `InnerTubeProfiles.swift`.

### Auto-update (Sparkle)

The app ships [Sparkle 2](https://sparkle-project.org) (SPM dependency, pinned `>= 2.9.0`) for in-place updates outside the Mac App Store. `UpdaterController` (`Services/UpdaterController.swift`) owns a single `SPUStandardUpdaterController`, started eagerly from `AppDelegate` so the automatic background check runs; the menu-bar "Check for Updates…" item calls it manually. Config lives in `Info.plist`: `SUFeedURL` → `https://github.com/TubeFold/App/releases/latest/download/appcast.xml`, `SUPublicEDKey` (the EdDSA public key), `SUEnableAutomaticChecks`. Hardened Runtime (Release-only) is required and already on; Sparkle.framework signs as part of the Developer ID archive/export.

**Releasing an update:** add a `## [<version>] - <date>` section to `CHANGELOG.md` (move items out of `## [Unreleased]`), bump `MARKETING_VERSION` + `CURRENT_PROJECT_VERSION`, run `scripts/release-macos.sh` (notarized zip), then `scripts/generate-appcast.sh` (EdDSA-signs the zip → `build/release/appcast.xml`). Upload **both** `TubeFold.zip` and `appcast.xml` to the GitHub release tagged `v<version>`.

**Release notes (single source = `CHANGELOG.md`):** each released version has a Keep-a-Changelog `## [X.Y]` section. `scripts/changelog.py` extracts that section's Markdown and (a) `generate-appcast.sh` embeds it into the appcast item as `<description sparkle:format="markdown">` (Sparkle 2.9+ renders Markdown directly — no HTML), so it shows in the in-app update dialog, and (b) CI passes it to `gh release create --notes-file` so the GitHub Release body is identical. Injection is idempotent and re-injection replaces the prior `<description>`. CI **fails the release** (guard step in `release.yml`) if no section matches the tagged version, so an appcast can't ship without notes; locally a missing section is non-fatal (just no notes). The **EdDSA private key** lives in the login keychain (created once via Sparkle's `generate_keys`) — back it up; losing it means no client can verify future updates. For CI, pass it as the `SPARKLE_ED_PRIVATE_KEY` secret (consumed by `generate-appcast.sh`).

**CI:** `.github/workflows/release.yml` automates all of the above on a `v*` tag push — imports the Developer ID cert into a temp keychain, runs `release-macos.sh` (notary auth via an Apple ID + app-specific password) + `generate-appcast.sh`, publishes the GitHub Release, then bumps `TubeFold/homebrew-tap` via `scripts/update-cask.sh`. It fails fast unless the tag equals `v<MARKETING_VERSION>`. Required secrets are listed at the top of the workflow (cert .p12 + password, `NOTARY_APPLE_ID` + `NOTARY_APP_PASSWORD`, Sparkle key, and `HOMEBREW_TAP_TOKEN` — a PAT with write access to the tap, since the default `GITHUB_TOKEN` is repo-scoped). `release-macos.sh` still also accepts a keychain profile or the App Store Connect API-key trio for local runs.

### Chrome extension

MV3 extension that POSTs the active YouTube tab to `http://127.0.0.1:43821`. It lives in its **own repo**, [TubeFold/extension](https://github.com/TubeFold/extension), with its own Chrome Web Store CI. The only contract between them is the localhost API port and the `chrome-extension://*` CORS origin; there is no build-time coupling.
