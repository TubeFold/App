# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

Tool that turns a YouTube URL into a Markdown summary. It does **not** use any LLM HTTP API — providers shell out to a locally-installed CLI driven by the user's own subscription and keep only the model's final message. Two providers ship: `codex` (the `codex` CLI, default) and `claude` (the `claude` / Claude Code CLI, via `claude --print`). The README is in Russian; this file and the code are in English.

## Commands

Run everything from the repo root — tests and the `tubefold` package assume the project root is the working directory / on `sys.path`.

```bash
# All tests (~90 of them; the canonical runner)
python3 -m unittest discover -s tests

# A single module / case / method
python3 -m unittest tests.test_youtube_url
python3 -m unittest tests.test_youtube_url.NormalizeYouTubeUrlTests.test_short_url

# Install: checks deps, seeds ~/.config/tubefold/config.env, symlinks both CLIs into ~/.local/bin
./install.sh

# Run the CLI without installing (the .venv python is used if present)
./bin/tubefold "https://youtu.be/dQw4w9WgXcQ" --verbose

# Exercise the full pipeline WITHOUT Codex (no model call) — use this for fast iteration
./bin/tubefold "https://youtu.be/dQw4w9WgXcQ" --provider fake --no-open

# Run the local HTTP backend by hand (only for debugging the API / extension; the macOS app launches it itself)
./bin/tubefold-server --provider codex   # or --provider fake

# Build the macOS app from the terminal
xcodebuild -project "TubeFold App/TubeFold.xcodeproj" \
  -scheme "TubeFold" -configuration Debug -destination "platform=macOS" build
```

`pyproject.toml` sets `[tool.pytest.ini_options] pythonpath = ["scripts"]`, so `pytest` also works, but `unittest discover` is what the project documents.

## Architecture

There are **two front ends over one shared pipeline**:

- **`bin/tubefold`** — one-shot, synchronous CLI. Self-contained orchestrator that runs the pipeline start-to-finish in a temp dir and prints the saved `.md` path to stdout (diagnostics go to stderr).
- **`bin/tubefold-server`** + the `tubefold/` package — a persistent localhost HTTP API with a SQLite store and a background job queue. Drives the Chrome extension and the macOS app.

Both reuse the same helpers in **`scripts/tubefold_lib.py`** (URL parsing, metadata-field extraction, `strip_outer_markdown_fence`, `validate_provider_response`, `yaml_front_matter`, safe/unique filenames) and the same pipeline shape:

```
URL → parse video id → (optional) yt-dlp metadata → youtube-transcript-api transcript
    → render prompt from prompts/<template>.md → provider → strip fence + validate → YAML front matter + body
```

Key invariant: **the YAML front matter is generated by the pipeline, never by the model.** The provider only ever produces the Markdown body, which is appended after the front matter.

### Provider contract

A provider is an executable resolved from `providers/<name>.sh` (or an absolute path), invoked as:

```
provider <prompt_file> <output_file>
```

It must write **only the final Markdown** to `<output_file>` and exit `0`. Config values are passed to the provider via environment variables (e.g. `CODEX_MODEL`, `CODEX_REASONING_EFFORT`, `CODEX_TIMEOUT_SECONDS`; `CLAUDE_MODEL`, `CLAUDE_REASONING_EFFORT`, `CLAUDE_TIMEOUT_SECONDS`).

- `providers/codex.sh` delegates to `scripts/run-codex-provider.py`, which runs `codex exec` from an **isolated temp dir**, reads the prompt from **stdin**, and captures the model's last message via `--output-last-message`. It also passes `--json` so Codex streams JSONL events to stdout. Codex runs with `--sandbox read-only --ephemeral --ignore-rules --skip-git-repo-check` so it can't touch the repo or network state.
- `providers/claude.sh` delegates to `scripts/run-claude-provider.py`, which runs `claude --print --model <m> [--effort <e>] --output-format json` from an **isolated temp dir** (so the CLI never discovers a CLAUDE.md), reads the prompt from **stdin**, and parses the model's text out of the JSON result object on **stdout**. Uses the user's Claude subscription (OAuth/keychain), not an API key — `--bare` is deliberately *not* used.
- `providers/fake.sh` emits canned Markdown (overridable via `$FAKE_PROVIDER_OUTPUT`). Use it for tests and any pipeline work that shouldn't burn a real provider run.
- **Usage sidecar:** both first-class wrappers parse token usage (and, for Codex, the live `rate_limits` quota) out of the CLI's JSON and write it to `<output_file>.usage.json` via `scripts/usage_sidecar.py` (`claude_usage_from_result` / `codex_usage_from_jsonl`). This keeps the provider contract intact — `<output_file>` still holds only the Markdown body. Capture is **best-effort**: a missing/garbled sidecar (e.g. the `fake` provider, or a CLI format change) simply means no usage is recorded, never a failed job. `processing.py._record_usage` reads the sidecar after a successful run and calls `repository.set_job_usage`.

To add a provider, drop an executable `providers/<name>.sh` honoring the contract — no other registration needed (though a *first-class* provider with onboarding/settings also needs a `ProviderDescriptor` in `provider_setup.py`; see below).

### Pipeline scripts (`scripts/`)

- `get-video-metadata.sh` — yt-dlp metadata; both orchestrators **fall back to a video-id-only stub** if yt-dlp is missing or fails (metadata is best-effort, never fatal).
- `fetch-transcript.py` / `youtube_transcript_source.py` — transcript fetch + language selection (manual preferred over auto-generated; honors `PREFERRED_TRANSCRIPT_LANGS` / `ALLOW_ANY_TRANSCRIPT_LANGUAGE`). A failed transcript **is** fatal.
- `render-prompt.py` — fills `prompts/<template>.md` with metadata + transcript.
- `embed-macos-backend.sh` — Xcode build-phase script; see below.

### CLI config layering (`bin/tubefold`)

Precedence, lowest to highest: `DEFAULT_CONFIG` in the script → `~/.config/tubefold/config.env` (or `--config` / `$TUBEFOLD_CONFIG`) → environment variables → CLI flags. Legacy keys `SUBTITLE_LANGS` / `ALLOW_ANY_SUBTITLE_LANGUAGE` are mapped onto the `*_TRANSCRIPT_*` keys, and the legacy `SUMMARY_LANGUAGE` key maps onto `OUTPUT_LANGUAGE`. Output filenames are derived from the video title, sanitized for macOS, and never overwrite — collisions get a ` (2)`, ` (3)` suffix. The summary's output language is configurable (`OUTPUT_LANGUAGE` / `--language`, default `English`): it is substituted into the prompt template as `{{OUTPUT_LANGUAGE}}` by `render-prompt.py`, so the template no longer hard-codes a language.

### `tubefold/` package (the local server)

Pure Python stdlib HTTP (`http.server.ThreadingHTTPServer`) — no web framework.

- `server.py` — request routing; bound to `127.0.0.1` only. Optional bearer auth via `TUBEFOLD_API_TOKEN` (disabled by default for local dev). `/health` advertises `apiVersion` and a `backendFeatures` map.
- `repository.py` — SQLite (`videos` + `jobs` tables) under the data dir. `create_or_reuse` does dedupe so the same YouTube video isn't re-summarized; `force_regenerate=True` overrides.
- `processing.py` — a single background worker thread draining queued jobs **one at a time**, writing per-stage logs and copying final artifacts into `videos/<youtube_id>/`. On startup it **reclaims orphaned jobs** (`reclaim_orphaned_jobs`) — any job left in a mid-run status by a previous crash/restart is failed with a synthetic error so it can't hang the queue.
- **Library delete** — `DELETE /api/v1/videos/{id}` → `repository.delete_video(id)` removes the row and returns the freed `youtube_video_id`. Surfaced in the macOS Library row and the extension popup.
- **Watch-activity suggestions** — the Chrome extension reports videos the user actually watches on YouTube via `POST /api/v1/watch-activity`; `repository.record_watch_activity` upserts them into a `watch_activity` table. `latest_watch_suggestion` returns the newest non-dismissed watched video that isn't already being/already summarized (joined against `videos`), so the app/extension can offer a "summarize this" nudge. `POST /api/v1/watch-activity/dismiss` (→ `dismiss_watch_activity`) hides a suggestion.
- `provider_setup.py` + `codex_settings.py` + `claude_settings.py` — CLI-provider onboarding, generalized over a `ProviderDescriptor` (one per provider: binary name, marker, model/effort options, connection-command builder). `CliProviderDiagnostics` does the shared work — detect the binary, run a marker connection test, persist setup state (state only — never credentials or full test output), validate model/effort choices — and `CodexProviderDiagnostics`/`ClaudeProviderDiagnostics` are thin subclasses. State keys are provider-prefixed (`codexExecutablePath`/`claudeExecutablePath`, `codexModel`/`claudeModel`, `codexConnectedAt`/`claudeConnectedAt`, …); `selectedProviderID` picks the active one. `ProviderSetupStore.select(provider)` switches it and recomputes the global `providerSetupCompleted`/`lastSuccessfulConnectionTest` from that provider's stored `*ConnectedAt`. `ProviderSetupStore` (`data_dir/provider-setup.json`) is also the app-settings store: it holds the `outputLanguage` setting (`output_language.py` normalizes it; default `English`), saved via `POST /api/v1/provider-setup/output-language` and returned in the `/api/v1/provider-setup` state. `processing.py` reads it and passes `--output-language` to `render-prompt.py`. Server endpoints are provider-parameterized: `POST /api/v1/provider-setup/{codex|claude}/{detect,test,model}`, `POST /api/v1/provider-setup/select` (`{provider}`), plus the shared `/complete` and `/output-language` which act on the selected provider; `GET /api/v1/provider-setup` returns the selected provider's state + model options and a `providers` list for the picker.
- **Which provider actually runs:** `processing.py`'s `_active_provider()` honors the UI selection — when `config.provider` is `codex`/`claude` it returns the store's `selectedProviderID` (if that provider's `providers/<id>.sh` exists), otherwise it returns `config.provider` verbatim (e.g. `fake`). So the embedded server is launched with `--provider codex` but the user's in-app provider choice drives each job **without a relaunch**.
- `telegraph.py` — "Publish to Telegraph" feature (stdlib only). Converts a stored summary's Markdown body into the Telegraph node DOM (`markdown_to_nodes`; headings collapse to `h3`/`h4` since Telegraph forbids `h1`/`h2`, content capped at 64 KB), talks to the public `telegra.ph` API via `urllib` (`TelegraphClient`, injectable `request_fn` so tests never hit the network), persists one anonymous account token created on first use (`TelegraphStore` → `data_dir/telegraph-account.json`), and publishes via `TelegraphPublisher`. One article per video: the URL is cached on the video row (`telegraph_url`/`telegraph_path`/`telegraph_summary_hash`); a repeat publish with an unchanged summary reopens the same URL, a regenerated summary updates the same page in place via `editPage`. Telegraph pages are **public and cannot be deleted** — only edited. Endpoint: `POST /api/v1/videos/{id}/publish-telegraph` → `{url, status}`. The macOS Library row and the Chrome extension popup both call it. The published article's source header also carries the read-time label (`build_article_content` calls `reading_time`) and the video's **publish date** (`📅 Published …`), parsed from the summary's `published_at` front matter via `front_matter_value` — never the processing date. (Telegraph's own byline date is the page-creation time and can't be set via the API, so the meaningful date lives in our header content.)
- `reading_time.py` — stdlib port of the [`readtime`](https://github.com/alanhamlett/readtime) algorithm (Medium's formula: 265 WPM, words split on `\W+`, inline-image time decaying 12s→3s, `max(1, ceil(seconds/60))` minutes). Reuses `telegraph.markdown_to_nodes` to extract the summary's visible text (link URLs dropped, labels kept). Surfaced two ways: `/api/v1/videos` includes `readingTimeMinutes` per ready video (computed on the fly in `_video_payload`), and the Telegraph header shows `"N min read"`.
- **Token-usage stats** — per-job token usage is stored on the `jobs` row (`provider`, `input_tokens`, `output_tokens`, `total_tokens`, `cost_usd`, and the Codex quota snapshot `weekly_percent`/`weekly_resets_at`/`primary_percent`, all added via `repository._migrate`). `repository.usage_summary()` aggregates them (total tokens, per-provider breakdown, and the freshest Codex weekly snapshot) and `GET /api/v1/usage` returns it. Counts cover **TubeFold's own runs only**. The macOS Settings "Usage" card (`UsageStatsView`) shows total/per-provider tokens; for **Codex** it draws a real weekly-quota gauge from `rate_limits.secondary.used_percent`, for **Claude** it shows tokens + cost (the Claude CLI doesn't report a weekly limit in `--print`).
- `config.py` — `AppConfig` dataclass, env-var driven. Data dir defaults to `~/Library/Application Support/TubeFold/` (`database.sqlite`, `videos/`, `jobs/`, `logs/`). `python_executable` defaults to `.venv/bin/python`.

### macOS app + backend compatibility check

The SwiftUI app (`TubeFold App/`) **embeds the entire Python backend and a framework Python runtime inside the `.app`** via the Xcode build phase that runs `scripts/embed-macos-backend.sh` (copies `bin/ tubefold/ scripts/ providers/ prompts/ config/ requirements.txt` + `Python.framework` + venv site-packages, rewrites the interpreter's install name, ad-hoc codesigns, and validates imports at build time). At runtime `BackendProcessController` launches and supervises the embedded `tubefold-server`; users never start it manually.

Because client and server ship together but can drift, `BackendProcessController` treats `/health` as a **compatibility gate**: it only reuses a running helper when `apiVersion == 1` **and every flag in `backendFeatures` is true**. Otherwise it kills the stale/incompatible helper and relaunches. **If you add a `backendFeatures` flag in `server.py`, update the Swift check in `BackendProcessController.swift` (and vice-versa) or the app will reject its own backend.** (Current flags: `codexModelSettings`, `libraryRegenerate`, `unlimitedTranscripts`, `telegraphPublish`, `outputLanguageSetting`, `readingTime`, `claudeProvider`, `usageStats`, `watchActivity`, `libraryDelete`.)

### Auto-update (Sparkle)

The app ships [Sparkle 2](https://sparkle-project.org) (SPM dependency, pinned `>= 2.9.0`) for in-place updates outside the Mac App Store. `UpdaterController` (`Services/UpdaterController.swift`) owns a single `SPUStandardUpdaterController`, started eagerly from `AppDelegate` so the automatic background check runs; the menu-bar "Check for Updates…" item calls it manually. Config lives in `Info.plist`: `SUFeedURL` → `https://github.com/TubeFold/App/releases/latest/download/appcast.xml`, `SUPublicEDKey` (the EdDSA public key), `SUEnableAutomaticChecks`. Hardened Runtime (Release-only) is required and already on; Sparkle.framework signs as part of the Developer ID archive/export — no extra release-script handling (unlike the embedded Python in `Contents/Resources`).

**Releasing an update:** bump `MARKETING_VERSION` + `CURRENT_PROJECT_VERSION`, run `scripts/release-macos.sh` (notarized zip), then `scripts/generate-appcast.sh` (EdDSA-signs the zip → `build/release/appcast.xml`). Upload **both** `TubeFold.zip` and `appcast.xml` to the GitHub release tagged `v<version>`. The **EdDSA private key** lives in the login keychain (created once via Sparkle's `generate_keys`) — back it up; losing it means no client can verify future updates. For CI, pass it as the `SPARKLE_ED_PRIVATE_KEY` secret (consumed by `generate-appcast.sh`).

**CI:** `.github/workflows/release.yml` automates all of the above on a `v*` tag push — imports the Developer ID cert into a temp keychain, runs `release-macos.sh` (notary auth via the App Store Connect API-key trio) + `generate-appcast.sh`, publishes the GitHub Release, then bumps `TubeFold/homebrew-tap` via `scripts/update-cask.sh`. It fails fast unless the tag equals `v<MARKETING_VERSION>`. Required secrets are listed at the top of the workflow (cert .p12, notary .p8 trio, Sparkle key, and `HOMEBREW_TAP_TOKEN` — a PAT with write access to the tap, since the default `GITHUB_TOKEN` is repo-scoped).

### Chrome extension (`chrome-extension/`)

MV3 extension that POSTs the active YouTube tab to `http://127.0.0.1:43821` (see `src/shared/constants.js`). Load unpacked from `chrome://extensions` for development.

## Notable constraints

- MVP scope: no playlists, no Whisper fallback, no chunked summarization (long transcripts are sent whole).
- The provider CLI (`codex` and/or `claude`) is an external user dependency even in the packaged app; only the Python backend is bundled. `install.sh` requires at least one of the two to be present.
